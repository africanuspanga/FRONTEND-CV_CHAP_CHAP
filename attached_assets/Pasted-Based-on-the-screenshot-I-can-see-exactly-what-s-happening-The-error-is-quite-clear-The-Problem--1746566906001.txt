How to Fix It
Here's my advice for the frontend team:

Solution 1: Use the Recommended API Format (Preferred)
Change your request format to separate template_id from CV data:

// Change from this:
{
  templateId: "jijengeclassic", 
  personalInfo: {...},
  // other fields
}
// To this:
{
  "template_id": "jijengeclassic",
  "cv_data": {
    "personalInfo": {
      "firstName": "John",
      "lastName": "Doe",
      "email": "john@example.com", // Make sure email exists!
      // other personal info
    },
    // Add these top-level fields for compatibility with all templates
    "name": "John Doe",
    "email": "john@example.com",
    // other cv data
  }
}
Solution 2: Keep Current Format but Add Required Fields
If you prefer keeping your current format with templateId inside the data:

{
  "templateId": "jijengeclassic",
  "personalInfo": {
    "firstName": "John",
    "lastName": "Doe",
    "email": "john@example.com" // Make sure this exists!
  },
  // Add these top-level fields (required by the API validation)
  "name": "John Doe", 
  "email": "john@example.com",
  // other cv data
}
Important Implementation Details
Always include top-level name and email:

The API validation specifically checks for these fields
You can derive them from personalInfo: name = personalInfo.firstName + " " + personalInfo.lastName
Ensure personalInfo has an email:

Even when providing a top-level email, the personalInfo.email is used by templates
The safest approach is to provide redundant data:

Include data in both formats (e.g., both workExperiences and experience)
Include data at both personalInfo level and top level
